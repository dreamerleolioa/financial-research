# AI Stock Sentinel 後續優化路徑 Roadmap

> 類型：高層規劃文件（Roadmap）
> 建立日期：2026-03-16
> 適用時機：完成 Analyze 頁「新倉策略建議」定位調整與新倉策略演算法升級後，作為下一輪需求拆解入口

---

## 1. 目的

本文件整理 AI Stock Sentinel 在「新倉策略建議」路線明確化之後的後續優化方向，目標不是直接取代 implementation plan，而是提供一張可持續拆解的中長期地圖。

後續每一項優化主題，原則上都應再獨立產出對應的需求文件或計劃文件，再進入實作。

---

## 2. 當前基線

截至目前，專案已完成或正在處理的關鍵前置工作如下：

1. `/analyze` 與 `/analyze/position` 的產品語義已切開
   - `/analyze`：新倉策略建議
   - `/analyze/position`：持股操作建議
2. Analyze 頁文案定位調整已有獨立計劃
3. 新倉策略演算法優化已有獨立計劃
4. Phase 8 路線中已包含歷史上下文與勝率回測相關基礎建設

因此，下一輪優化不應再回到「命名混亂」或「頁面定位不清」的問題，而應集中在可靠度、校準、可觀測性與產品體驗深化。

---

## 3. Roadmap 原則

後續優化建議遵循以下順序原則：

1. 先補可驗證性，再補更複雜的判斷
2. 先補資料與策略校準，再補更多 UI 包裝
3. 先補系統可觀測性，再做權重調優
4. 讓每次策略升級都能被回測、測試與版本資訊追蹤

換句話說，下一輪最重要的不是把畫面做得更豐富，而是把「這個系統為什麼值得相信」補完整。

## 5. 各主題說明

### 5.0 P0 前置：回測腳本基礎建設

**為什麼必須優先：**

P0 的兩項工作（回測與信心校準）都依賴回測腳本，但若腳本輸出格式不穩定、或對「勝率」「進場成功」的定義沒有統一，不同人寫出來的腳本會量到不同的東西，校準結果也沒有意義。這項工作是 P0 的地基。

**建議方向：**

1. 確認現有 CLI 回測腳本的輸出格式，並固定為標準 schema
2. 統一指標定義：
   - 「進場成功」的定義（例如持有 N 日後報酬超過基準）
   - 持有週期：5 / 10 / 20 交易日
   - 報酬計算：絕對報酬 or 相對大盤
   - 勝率計算邏輯（含部分勝算？平手如何歸類？）
3. 確認腳本可穩定跑完並輸出結果（無靜默失敗、無缺漏欄位）
4. 輸出格式與指標定義列入 P0 需求文件的必要驗收條件

**可拆出的需求文件主題：**

1. 回測腳本輸出格式標準化需求
2. 策略評估指標定義文件

**依賴：**

1. 現有回測腳本可執行
2. 歷史策略輸出資料可取得

---

### 5.0b P0 前置：策略版本號佔位

**為什麼必須優先：**

策略版本化若延遲到 P2，之前累積的所有回測與校準結果都不會有版本標記，之後要做比較或回歸分析時，無法確認當時使用的是哪版邏輯，前期工作的可追溯性會大幅下降。

**建議方向：**

1. 在開始 P0 回測之前，定義最簡版本號規範（例如一個 config 常數或 `strategy_version` 欄位）
2. 不需要完整的版本化系統，只需確保每次回測結果都帶有可識別的版本標記
3. 完整的策略版本化功能仍留在 P2，這裡只做「佔位」

**可拆出的需求文件主題：**

1. 策略版本號最簡規範（輕量）

**依賴：**

1. 無（主動前置，成本極低）

---

### 5.0c P0 前置：資料源品質監控

**為什麼提前至 P0 前置：**

原本排在 P1，但若資料源有 fallback 問題或欄位缺失，P0 的回測資料本身就是髒的，用髒資料校準出來的信心分數沒有任何意義。因此資料源的基本可觀測性必須在回測開始前先到位。

**建議方向：**

1. 記錄各資料源成功率、fallback 命中率、欄位缺失率
2. 記錄平均延遲與超時率
3. 讓問題定位到 provider 層，而不是統一歸咎「AI 不準」
4. P0 前置階段只需基本 logging，完整 dashboard 可在 P1 補完

**可拆出的需求文件主題：**

1. Provider health monitoring 需求（基礎版）
2. Data source observability 需求

**依賴：**

1. 無（主動前置）

---

### 5.1 P0：新倉策略回測與校準

**為什麼優先：**

新倉策略即使做完 evidence-based 升級，如果沒有歷史回測，仍然只是「看起來合理」。回測是把策略從主觀說服力提升到客觀可驗證性的第一步。

**建議方向：**

1. 以策略輸出為單位，觀察後續 5 / 10 / 20 交易日報酬與勝率
2. 依 `strategy_type`、`conviction_level`、`confidence_score` 分箱
3. 觀察不同 market regime 下的策略表現差異
4. 與現有 CLI 腳本和 Phase 8 回測能力銜接

**可拆出的需求文件主題：**

1. 新倉策略標籤回測需求
2. 新倉策略勝率/報酬分箱分析需求
3. 新倉策略回測輸出格式與報表需求

**依賴：**

1. 新倉策略演算法升級完成
2. 策略輸出結構穩定

---

### 5.2 P0：信心分數校準

**為什麼優先：**

如果 `confidence_score` 不能可靠反映訊號品質，那它會傷害整體產品信任感。高分不準，比沒有分數更糟。

**建議方向：**

1. 分析不同 confidence 區間的真實表現
2. 驗證 `confidence_score` 與後續勝率/報酬是否正相關
3. 必要時調整 `confidence_scorer.py` 權重或分數映射方式
4. 與策略輸出一起看，而不是單獨看分數

**可拆出的需求文件主題：**

1. 信心分數校準需求
2. confidence 與勝率對照需求
3. 半自動調權流程需求

**依賴：**

1. 回測資料足夠
2. 勝率回測腳本可穩定輸出結果

---

### 5.3 P1：Analyze 頁策略卡升級

**為什麼要做：**

後端就算輸出更多 evidence，若前端仍只顯示四個欄位，使用者仍會覺得策略很薄。

**建議方向：**

1. 將策略卡改成四段式結構：建議動作、主要理由、關鍵價位、失效條件
2. 顯示 `conviction_level`
3. 顯示 2–3 條 `thesis_points`
4. 顯示主要 `invalidation_conditions`
5. 清楚區分「事實性價位」與「推論性判斷」

**可拆出的需求文件主題：**

1. Analyze 頁策略卡重構需求
2. 新倉策略 explainability UI 需求

**依賴：**

1. action_plan 結構擴充完成

---

### 5.4 P1：盤中 vs 收盤策略分流

**為什麼要做：**

盤中資料未定稿，若與收盤版使用相同策略積極度，容易誤導使用者追價或過度解讀訊號。

**建議方向：**

1. 盤中版限制最大 `conviction_level`
2. 盤中版限制建議首筆部位大小
3. 收盤後允許完整策略級別與較完整結論
4. 把 `is_final` 真正納入策略 guardrails，而不只用於免責聲明

**可拆出的需求文件主題：**

1. 盤中/收盤策略分流需求
2. 盤中策略安全降級需求

**依賴：**

1. API 快取與 `is_final` 路徑穩定

---

### 5.5 P1：資料源品質監控（完整版）

**說明：**

基礎版已提前至 P0 前置（見 5.0c）。P1 階段的目標是在基礎 logging 的基礎上，補完整的可觀測性工具，例如 dashboard 或告警機制。

**建議方向：**

1. 建立 provider health dashboard，匯總各資料源的成功率趨勢
2. 設定異常告警（fallback 命中率超過閾值時通知）
3. 與回測結果串接，讓回測報告能標注「此段期間資料品質」
4. 讓問題定位到 provider 層，而不是統一歸咎「AI 不準」

**可拆出的需求文件主題：**

1. Provider health dashboard 需求
2. Data source observability 完整版需求

---

### 5.6 P2：LLM 輸出評測

**為什麼要做：**

雖然策略本身是 rule-based，但使用者最終看到的綜合敘事仍高度受 LLM 影響。如果 LLM 過度樂觀、混淆維度、或把 rule-based 結論講錯，體驗仍會崩。

**建議方向：**

1. 評測 `analysis_detail` 是否越界引用不該混入的維度
2. 檢查 `final_verdict` 是否和 rule-based 策略相衝突
3. 檢查過度肯定、過度武斷的措辭
4. 建立少量固定案例做 regression eval

**可拆出的需求文件主題：**

1. LLM 分析輸出評測需求
2. analysis_detail regression eval 需求

---

### 5.7 P2：策略版本化（完整版）

**說明：**

版本號佔位已提前至 P0 前置（見 5.0b），確保 P0 回測結果具備可追溯性。P2 階段的目標是在佔位基礎上，建立完整的版本化系統。

**建議方向：**

1. 為策略輸出正式定義 `strategy_version` schema
2. 在 cache / log / backtest 中保留完整版本資訊
3. 回測時可依策略版本切分比較、做回歸分析
4. 策略規則更新時自動遞增版本號

**可拆出的需求文件主題：**

1. 策略版本化完整需求
2. 分析結果版本追蹤需求

---

### 5.8 P3：新倉/持股體驗整合

**為什麼放較後：**

這是體驗加值項，但要先建立在策略語義清楚、演算法穩定、輸出可解釋之上。

**建議方向：**

1. 在持股列表中能連動到新倉視角與持股視角
2. 同一標的提供兩種視角，但明確分欄，不混語意
3. 支援從 Portfolio flow 直接查看該標的的 Analyze 結果

**可拆出的需求文件主題：**

1. Portfolio 與 Analyze 決策體驗整合需求
2. 新倉/持股雙視角 UI 需求

## 6. 建議拆解順序

建議你後續依這個順序逐一產出需求文件：

**P0 前置（先做，作為後續地基）**

1. 回測腳本基礎建設（輸出格式標準化 + 指標定義統一）
2. 策略版本號佔位（最簡版，成本極低）
3. 資料源品質監控（基礎版 logging）

**P0（前置完成後）**

4. 新倉策略回測與校準
5. 信心分數校準

**P1**

6. Analyze 頁策略卡升級
7. 盤中 vs 收盤策略分流
8. 資料源品質監控（完整版 dashboard）

**P2**

9. LLM 輸出評測
10. 策略版本化（完整版）

**P3**

11. 新倉/持股體驗整合

這個順序的核心考量是：

1. 先建立回測地基（格式、指標、資料品質、版本佔位），讓 P0 的工作不虛耗
2. 再驗證策略可靠度
3. 再調整前台呈現
4. 最後做體驗串接與擴張

後續每一項主題，建議至少拆成以下其中一種文件：

1. 需求文件

## 8. 不建議現在就做的事

以下項目目前不建議優先投入：

1. 一開始就自動調整權重到生產環境
2. 在沒有版本化前就大量比較不同策略邏輯
3. 在沒有回測證據前，先做大量視覺包裝
4. 把新倉與持股邏輯重新混在同一張策略卡

---

## 9. 結論

此專案下一輪最值得投入的方向，不是再多做一層新功能，而是把策略系統補成「可驗證、可校準、可觀測、可解釋」。

當前最合理的後續節奏是：

1. 先建立回測地基（腳本格式、指標定義、資料品質監控、版本號佔位）
2. 再做回測與信心校準
3. 再做策略卡 explainability 升級與盤中/收盤分流
4. 再做資料源完整可觀測性與 LLM 輸出評測
5. 最後才做跨頁體驗整合

本文件作為後續需求拆解入口使用，不直接作為單一開發任務的實作文件。
